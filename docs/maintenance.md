# 清理与维护策略

## 当前自动清理能力

当前项目没有“自动删除代码”的逻辑，也不建议默认启用自动删除。

原因：

- 项目还在快速演进阶段，失败代码、重复代码、临时代码有时具有诊断价值
- 自动删除可能破坏可回溯性
- spec-coding 更适合“先生成清理候选清单，再人工确认”

## 已有防护

| 机制 | 作用 |
| --- | --- |
| `.gitignore` | 忽略 `.DS_Store`、`target/`、`.env`、本地下载等 |
| `docs/cleanup_candidates.md` | 清理候选清单与理由 |
| `tests/run_local.sh` | 清理/重构后验证项目是否仍可运行 |
| `docs/progress.md` | 防止误删当前阶段需要的文件 |
| `docs/CONTEXT_HANDOFF.md` | 上下文重置后快速恢复项目状态 |

## 推荐清理流程

每次清理都按这个流程：

1. 只读扫描当前文件和 git 状态
2. 生成待清理清单
3. 对每个候选项写明原因和风险
4. 等用户确认
5. 删除确认范围内的文件
6. 按风险选择验证命令；结构清理优先运行 `./tests/run_local.sh`，纯构建缓存清理可先做只读核对
7. 同步更新文档：`docs/cleanup_candidates.md` 记录执行结果，`docs/progress.md` 记录状态变化，`docs/DOCUMENT_MAP.md` 更新路径入口，必要时更新 `README.md` 和 `docs/CONTEXT_HANDOFF.md`

## 可清理类型

| 类型 | 默认策略 |
| --- | --- |
| `.DS_Store` | 可直接列为推荐清理 |
| `src/backend/target/` | 可列为推荐清理，但测试后会重新生成 |
| 临时下载目录 | 可列为推荐清理，前提是结果已经记录 |
| 未选中的 demo | 用户确认后删除 |
| 视觉参考 demo | 用户确认后优先移入 `docs/design/`，再更新文档入口 |
| 重复代码 | 先标记，再重构，不直接删 |
| 失败代码 | 如果没有测试覆盖，先保留或迁移到 issue/文档说明 |
| 臃肿代码 | 通过 review 找出，拆小或删减，必须跑测试 |

## 定时清理建议

如果要设置自动化，建议只做 **定时清理审计**，不要自动删除：

- 每周扫描一次
- 输出清理候选清单
- 标记重复/失败/臃肿代码风险
- 不修改文件
- 等用户确认后再删除

可选频率：

- 每周一次：适合当前阶段
- 每个里程碑结束后：更符合 spec-coding
- 手动提示：最安全，当前也足够

## 当前建议

现阶段建议先采用手动清理提示。

当前已经进入 Axum API 与前端维护阶段，清理审计仍建议手动触发或每个里程碑结束后触发。自动化内容应是生成报告，不是直接删除代码。
